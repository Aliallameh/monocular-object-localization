Real-time fixed-camera pipeline that detects a standard outdoor garbage bin, estimates its position in the world frame, and streams coordinates to stdout — frame by frame, under 150 ms cold-start on CPU.

```bash
bash run.sh --video input.mp4 --calib calib.json
```

> `input.mp4`, `calib.json`, and `waypoints.json` are runtime assets and are not committed. All generated CSV/plots/JSON are regenerated by the command above.

## 📐 Localization

### Coordinate frames

```
Camera (OpenCV)          World (pole base)
  +X  right                +X  forward
  +Y  down                 +Y  left
  +Z  optical axis         +Z  up
```

`calib.json` provides `K`, `dist_coeffs`, `camera_height_m`, `camera_tilt_deg`. The transform is:

```
P_world = R_cw @ P_cam + t_cw       t_cw = [0, 0, camera_height_m]
```

### Estimator 1 — height-based centroid

Classic pinhole depth from known bin height `H = 0.65 m`:

```
Z = fy * H / h_px
X = x_norm * Z
Y = y_norm * Z
```

Logged per frame in `results/diagnostics.csv`.

### Estimator 2 — ground-contact centroid *(default)*

Bottom-center pixel ray-cast to the flat ground plane, lifted by `H/2`:

```
r_cam      = normalize([x_norm, y_norm, 1])
r_world    = R_cw @ r_cam
lambda     = -camera_height_m / r_world_z
P_ground   = t_cw + lambda * r_world
P_centroid = P_ground + [0, 0, H/2]
```

The fixed-camera + flat-floor constraint is tighter than pixel-height alone, so this is the default output. Both estimators run in parallel for diagnostics.

**Synthetic validation** (from calibration, not video GT):

| Check | Value |
|---|---:|
| Ground-contact RMSE | 0.0049 m |
| Ground-contact max error | 0.0090 m |
| Height-based RMSE | 0.0907 m |
| Height-based max error | 0.1696 m |
| Median Δ between estimators on video | 0.300 m |
| p95 Δ between estimators on video | 0.709 m |

## 🎛️ Kalman Filter

Constant-velocity world-position filter:

```
state       x = [x, y, z, vx, vy, vz]
measurement z = [x, y, z]
```

`F` uses standard constant-velocity kinematics with `dt = 1/fps`. `Q` is white-noise acceleration discretization. Measurement variance `R` is adapted per detector source. Covariance update uses Joseph form.

Default config (`configs/default.yaml`):

```yaml
kalman:
  process_var: 3.0
  measurement_var: 0.01
```

Smoothing evidence (replay on this clip — no independent world GT exists):

| Metric | Value |
|---|---:|
| Frame-step std reduction | 59.9% |
| Second-difference std reduction | 84.4% |

Grid search (`experiments/kalman_gridsearch.py`, 24 configs) finds the max-smoothing point at `process_var=0.3 / measurement_var=0.1` with 74.1% frame-step reduction. The default stays at `3.0 / 0.01` — fitting smoothing on a single clip without a held-out set is just overfitting to the scene.

## 🛡️ Occlusion Handling

Three-layer dropout recovery:

1. BBox Kalman prediction keeps the track alive when detection drops out
2. Lucas-Kanade sparse optical flow propagates the bbox
3. Stdout emits `OCCLUDED ... age=N` for held tracks

In the delivered clip the bin stays visible to the blue-mask detector the entire time, so there are zero real occluded rows in `results/output.csv`. The code path is validated via a forced-dropout stress suite:

| Scenario | Suppressed frames | Continuity | Mean IoU vs normal | Min IoU |
|---|---:|---:|---:|---:|
| mid_stop_dropout | 26 | 100.0% | 0.766 | 0.600 |
| moving_crossing_dropout | 14 | 100.0% | 0.675 | 0.578 |
| late_low_confidence_dropout | 15 | 100.0% | 0.909 | 0.890 |

## 🛰️ Jetson Orin NX / UAV Deployment

Current run is CPU-only / classical. Notes for dropping in a learned detector and flying:

| Topic | Note |
|---|---|
| Quantization | TensorRT FP16 first. INT8 needs representative field frames for calibration or you'll see accuracy regression. |
| Toolchain | TensorRT = NVIDIA Jetson. RKNN = Rockchip (RK3588). Don't mix them up. |
| Moving platform | Swap fixed `R_cw, t_cw` for `T_world_body(t) @ T_body_cam`, where `T_world_body` comes from the vehicle EKF timestamped to match the frame. |
| IMU fusion | Vehicle EKF state: `[p, v, q, b_gyro, b_acc]`. Camera detections enter as **delayed position measurements** via the EKF update with their estimated covariance — never as direct truth overwrites. |
| Target filter | The bin KF `[x, y, z, vx, vy, vz]` runs in the EKF world frame. `t_cw` must be updated every frame from the current EKF pose — not held fixed. |
| MAVLink | `LANDING_TARGET` (#158) for target-relative tracking; `VISION_POSITION_ESTIMATE` (#102) for vehicle pose to flight controller. |
| Publish rate | 10–20 Hz is realistic. Always send position covariance with the estimate. |

**Latency budget on Orin NX:**

| Stage | Budget |
|---|---:|
| Capture | 5–12 ms |
| Detect (TensorRT FP16) | 8–25 ms |
| Localize | < 1 ms |
| Kalman update | < 1 ms |
| MAVLink / UART | 1–3 ms |
| **End-to-end** | **30–55 ms** |

## ⚠️ Known Limitations

- Detector is tuned to this clip (blue-masked bin, controlled lighting). Not a general outdoor trash-bin detector out of the box.
- Occluded-frame IoU is 73.3% — the blue-mask bbox sometimes grows to include floor reflection or stale lower-bin pixels near partial occlusions.
- No independent world-coordinate GT. Kalman analysis is smoothing/jitter analysis only.
- `z_world` is fixed by the ground-plane model at `H/2 = 0.325 m`. Sloped terrain or bottom-edge truncation will bias it.
