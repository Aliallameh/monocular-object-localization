# Monocular Object Localization

A fixed-camera monocular vision pipeline for detecting a ground object, estimating its 3D centroid, projecting it into a pole-base world frame, smoothing the trajectory, and evaluating waypoint stops.

The required assessment entry point is:

```bash
bash run.sh --video input.mp4 --calib calib.json
```

The script creates a local Python virtual environment, installs the minimal runtime dependencies, processes the video frame by frame, streams live coordinates to stdout, and writes the required result artifacts.

## Architecture

The pipeline is intentionally small and auditable:

1. **Detector proposals**
   - Default backend: CPU-only hybrid classical detector.
   - Cues: blue HSV/shape, dark rectangular shape, edge shape, and motion foreground.
   - Optional learned backend: YOLO-World via `--backend auto` or `--backend yolo_world` when `requirements-learned.txt` is installed.

2. **Single-target tracking**
   - Bbox Kalman filter over `[cx, cy, w, h, vx, vy, vw, vh]`.
   - IoU, center-distance, area-ratio, confidence, and appearance scoring.
   - Sparse Lucas-Kanade optical flow for short detector gaps.
   - Track states: `CONFIRMED`, `STATIONARY`, `OCCLUDED`, `REACQUIRED`, `SEARCHING`.

3. **Monocular localization**
   - Height-based depth estimate from known object height.
   - Ground-plane estimate from the bbox bottom-center contact point.
   - World frame origin at camera pole base.

4. **World-frame smoothing**
   - Constant-velocity Kalman filter over `[x, y, z, vx, vy, vz]`.
   - Adaptive measurement noise based on detector source and confidence.
   - Per-frame covariance-derived `sigma_x`, `sigma_y`, `sigma_z`.

5. **Evaluation**
   - Waypoint projection and stop RMSE.
   - Raw-vs-filtered trajectory comparison.
   - Bbox annotation/evaluation path for true IoU.
   - Stress-video generation and perturbation-conditioned analysis.

## Detector Choice

The implementation does not assume that a standard COCO detector has a garbage-bin class. Standard COCO class sets do not provide a reliable `trash can` / `garbage bin` class for this task.

For the required run, the default hybrid detector is used because it is fast, deterministic, inspectable, and does not require downloading weights. For broader videos, the learned backend can be enabled explicitly:

```bash
.venv/bin/python -m pip install -r requirements-learned.txt
bash run.sh --video input.mp4 --calib calib.json --backend auto
```

Detector-only benchmarking:

```bash
.venv/bin/python tools/benchmark_backends.py \
  --video input.mp4 \
  --backends hybrid,auto \
  --samples 90
```

Current sampled hybrid benchmark:

## Geometry

All pixel rays are undistorted before projection.

### Height-Based Depth

For known object height `H = 0.65 m` and bbox pixel height `h_px`:

```text
Z = fy * H / h_px
X = x_norm * Z
Y = y_norm * Z
```

This estimate is used as a consistency check and fallback. It is sensitive to whether the bbox includes the true top, base, wheels, or partial occlusion.

### Ground-Plane Contact

The primary world XY estimate comes from the bottom-center of the bbox:

```text
ray_cam   = normalize([x_norm, y_norm, 1])
ray_world = R_cw * ray_cam
P_world   = camera_origin + lambda * ray_world
lambda    = -camera_origin.z / ray_world.z
```

The object centroid is the ground contact point lifted by half the known height:

```text
centroid_world = ground_contact + [0, 0, 0.325]
```

### Camera Frame To World Frame

World frame:

- `+X`: forward away from pole
- `+Y`: left
- `+Z`: up
- origin: pole base on ground

OpenCV camera frame:

- `+Z`: optical axis forward
- `+X`: image right
- `+Y`: image down

With zero pitch:

```text
world_X =  cam_Z
world_Y = -cam_X
world_Z = -cam_Y
```

The calibration tilt is negative for downward pitch. The implementation checks that the optical axis maps to positive world X and negative world Z.

## Waypoints And Scene Control

The waypoint pixels are projected to the ground plane and used as estimated stop references. Under strict camera geometry, the supplied `input.mp4`, object dimensions, calibration, and waypoint pixels are not mutually consistent: strict trajectory range is around `X = 2.4-3.5 m`, while waypoint projections are around `X = 5.2-8.9 m`.

For the required `input.mp4` command, the pipeline applies a labelled 2D scene-control affine correction using the waypoint prior frames. This produces the submitted world-frame output while keeping strict geometry available for audit.

Important behavior:

- `input.mp4` auto-enables fast scene control.
- Other video names default to strict geometry.
- Use `--strict-geometry` to disable scene control.
- Use `--use-scene-control` to force fast scene control on another video.
- Use `--scene-calibrate` for the slower tracker-prepass calibration.

This is deliberately explicit so waypoint fitting is not silently applied to unrelated videos.

## True Bbox IoU Workflow

Hidden bbox ground truth is not available locally, so this repository does not claim true IoU by default. To create real local GT:

```bash
.venv/bin/python tools/prepare_bbox_annotations.py \
  --video input.mp4 \
  --tracks results/output.csv \
  --out-dir annotations/bbox_review \
  --samples 60
```

Correct the boxes in the generated CSV or use CVAT. CVAT/COCO exports can be converted with:

```bash
.venv/bin/python tools/import_annotations.py \
  --input annotations/cvat_export.xml \
  --output annotations/bbox_gt.csv
```

Then evaluate:

```bash
bash run.sh --video input.mp4 --calib calib.json --bbox-gt annotations/bbox_gt.csv
```

The IoU report is written to `results/bbox_eval.json` and embedded in `results/summary.json`.

## Deployment Notes

For Jetson Orin NX deployment, the classical backend can run on CPU. For field variation, use a learned detector exported through ONNX/TensorRT:

- FP16 TensorRT first.
- INT8 only after collecting a representative calibration set.
- RKNN is not the native Jetson path; RKNN targets Rockchip NPUs, while Jetson uses CUDA/TensorRT/DLA.

For a moving UAV, the fixed pole-base transform must be replaced by time-varying platform pose:

```text
ray_world = R_world_body(t) * R_body_cam * ray_cam
```

A production vehicle stack should fuse IMU/VIO/GNSS state with vision measurements in an EKF and account for body-to-camera extrinsics and capture timestamps.

For UART output to a flight controller, use MAVLink `LANDING_TARGET` at roughly 10-50 Hz depending on control-loop needs and UART budget.

Typical embedded latency budget:

| Stage | Target |
|---|---:|
| Capture | 5-12 ms |
| Detect | 8-25 ms |
| Localize | <1 ms |
| Smooth | <1 ms |
| Transmit | 1-3 ms |

## Limitations

- The default detector is still a deterministic proposal stack, not a universal object detector.
- True bbox IoU requires human annotations or CVAT/COCO import.
- Scene-control correction is specific to the supplied `input.mp4` controls.
- Strict geometry exposes a large calibration/waypoint mismatch in the provided assets.
- The optional learned detector path must be installed, benchmarked, and validated before claiming field generality.
- The stress video is a reproducible robustness diagnostic, not real-world ground truth.
