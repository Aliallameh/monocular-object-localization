# Monocular Garbage-Bin Tracking and Localization

Real-time fixed-camera pipeline for detecting a standard outdoor garbage bin,
estimating its camera/world-frame centroid, and streaming frame-by-frame
coordinates to stdout.

```bash
bash run.sh --video input.mp4 --calib calib.json
```

`input.mp4`, `calib.json`, and `waypoints.json` are runtime assets. They are
not committed and are not read from a hidden copy. Generated CSV/plots/JSON are
submission evidence and are regenerated by the command above.

## Quick Start

```bash
git clone https://github.com/Aliallameh/monocular-object-localization.git
cd monocular-object-localization
bash run.sh --video input.mp4 --calib calib.json
.venv/bin/python tools/validate_submission.py
.venv/bin/python -m unittest discover -s tests -v
```

The first tracked stdout line appears in about 89 ms on the regenerated local
run (`results/summary.json:first_track_stdout_ms_from_python`).

## Pipeline

```text
frame -> detector -> bbox tracker / LK flow -> localizer -> Kalman smoother -> stdout + CSV
```

| Module | File | Role |
|---|---|---|
| Detector | `detector.py` | Classical multi-cue bin proposals |
| Tracker | `tracker_utils.py` | BBox Kalman association and optical-flow bridge |
| Geometry | `localizer.py` | Camera/world transform, two monocular estimators |
| Plot/QA | `plots.py`, `qa_utils.py` | Trajectory, diagnostics, QA JSON |

No YOLOv8 bin-class accuracy claim is made. COCO YOLOv8 does not provide a
dedicated garbage-bin/trash-can class, and the optional YOLO baseline script is
only an any-object speed/coverage sanity check unless external bbox/class ground
truth is supplied.

## Real GT and Occlusion Evidence

Do not use generated tracker output as ground truth. The supported GT path is a
reviewed external annotation file, preferably CVAT/COCO export or a corrected
CSV. The evaluator accepts `occluded` labels and reports visible/occluded IoU
and continuity separately.

CSV schema:

```text
frame_id,x1,y1,x2,y2,review_status,occluded,visibility,bbox_type
```

Use `review_status=ok` for reviewed rows and `skip` for ambiguous frames. For
real occlusion evidence, mark `occluded=1`. If the bin extent is inferable
behind a person, use `bbox_type=amodal_full_bin`; if only the visible portion is
annotated, set `bbox_type=visible_extent` and interpret IoU accordingly.

Fast annotation workflow:

```bash
bash run.sh --video input.mp4 --calib calib.json
.venv/bin/python tools/prepare_bbox_annotations.py \
  --video input.mp4 \
  --tracks results/output.csv \
  --out-dir annotations/real_gt_review \
  --samples 80 \
  --include-ranges 420:470
```

Correct the draft boxes in `annotations/real_gt_review/bbox_annotator.html`,
download/save the corrected CSV as `annotations/bbox_gt.csv`, then run:

```bash
bash run.sh --video input.mp4 --calib calib.json --bbox-gt annotations/bbox_gt.csv
.venv/bin/python tools/validate_submission.py \
  --require-bbox-gt \
  --require-occlusion-gt \
  --min-gt-frames 30 \
  --min-gt-occluded-frames 5
```

CVAT/COCO import path:

```bash
.venv/bin/python tools/import_annotations.py \
  --input annotations/export.xml \
  --output annotations/bbox_gt.csv \
  --label bin
bash run.sh --video input.mp4 --calib calib.json --bbox-gt annotations/bbox_gt.csv
```

GT results are written to `results/bbox_eval.json` and embedded in
`results/summary.json:bbox_evaluation`. Key fields are:

| Field | Meaning |
|---|---|
| `all_frames.iou_over_0_6_rate` | IoU pass rate over reviewed GT frames |
| `visible_frames.*` | Metrics only on `occluded=0` rows |
| `occluded_frames.continuity_rate` | Tracker produced a row on occluded GT frames |
| `occluded_frames.iou_over_0_6_rate` | Box accuracy on occluded GT rows |
| `detector_source_counts_on_gt_occluded` | Whether occluded GT was handled by detector, LK flow, or prediction |

### Manual Floor-Contact Validation

`manual_contact_points.csv` contains three human-reviewed floor-contact pixels:
the point where the bin actually touches the ground in the image. This is not
bbox GT and not occlusion GT. It validates the ground-contact geometry only, and
the pipeline never uses these points to calibrate, correct, or generate
`results/output.csv`.

Run with:

```bash
bash run.sh --video input.mp4 --calib calib.json --contact-gt manual_contact_points.csv
```

Results are written to `results/contact_point_eval.json` and embedded in
`results/summary.json:contact_point_evaluation`.

| Frame | Manual contact px | Detected bottom-center px | Pixel error | Ground XY error |
|---:|---:|---:|---:|---:|
| 50 | (922, 885) | (977.1, 879.8) | 55.3 px | 0.110 m |
| 200 | (576, 694) | (587.0, 695.8) | 11.1 px | 0.036 m |
| 500 | (462, 760) | (469.6, 827.0) | 67.4 px | 0.379 m |
| Mean | - | - | 44.6 px | 0.175 m |

Interpretation: frame 200 supports the current contact geometry; frames 50 and
especially 500 expose that the detector's bbox bottom edge can follow the
visible wheel/body extent instead of the true floor-contact point. This is a
real limitation, not an accuracy claim. These reviewed points are useful for
tuning the detector/contact estimate later, but they are deliberately not used
as a correction layer in this submission.

## Localization

### Coordinate Frames

```text
OpenCV camera frame          World frame fixed at pole base
  +X right                     +X forward from pole
  +Y down                      +Y left
  +Z optical axis forward      +Z up
```

`calib.json` supplies `K`, `dist_coeffs`, `camera_height_m`,
`camera_tilt_deg`, and image size. The code builds:

```text
P_world = R_cw @ P_cam + t_cw
t_cw = [0, 0, camera_height_m]
```

For a level camera, the axis swap maps `+Z_cam -> +X_world`,
`+X_cam -> -Y_world`, and `+Y_cam -> -Z_world`. Downward pitch is then applied
about world Y. A runtime sign check verifies the optical axis points forward and
down.

### Two Estimators

`localizer.py` exposes both estimators explicitly.

1. `height_based_centroid`

Direct known-dimensions pinhole depth from bin height (`H = 0.65 m`):

```text
Z = fy * H / h_px
X = x_norm * Z
Y = y_norm * Z
```

This satisfies the assessment's known-object-dimension distance requirement and
is logged per frame in `results/diagnostics.csv`.

2. `ground_contact_centroid`

Bottom-center bbox pixel is undistorted, ray-cast to the flat ground plane, and
lifted by `H/2`:

```text
r_cam      = normalize([x_norm, y_norm, 1])
r_world    = R_cw @ r_cam
lambda     = -camera_height_m / r_world_z
P_ground   = t_cw + lambda * r_world
P_centroid = P_ground + [0, 0, H/2]
```

The ground-contact estimator is the default world-output method because the
fixed-camera/flat-floor constraint is stronger than bbox height alone. The
height estimator remains logged as an independent diagnostic. No waypoint or
scene-control calibration is applied to output coordinates.

Synthetic camera-model validation, generated from the supplied calibration and
not from video ground truth:

| Check | Value |
|---|---:|
| Ground-contact synthetic RMSE | 0.0049 m |
| Ground-contact max synthetic error | 0.0090 m |
| Height-based synthetic RMSE | 0.0907 m |
| Height-based max synthetic error | 0.1696 m |
| Median height-vs-ground XY delta on video | 0.300 m |
| p95 height-vs-ground XY delta on video | 0.709 m |

These numbers validate implementation consistency, not real-world accuracy.

## Kalman Filter

The world-position filter is a constant-velocity Kalman filter:

```text
state x = [x, y, z, vx, vy, vz]
measurement z = [x, y, z]
```

`F` uses the standard constant-velocity transition with `dt = 1/fps`. `Q` uses
white-noise acceleration discretization, and the covariance update uses Joseph
form. Measurement variance is adapted by detector source/confidence.

Runtime defaults (`configs/default.yaml`):

```yaml
kalman:
  process_var: 3.0
  measurement_var: 0.01
```

There is no independent world-coordinate ground truth for the bin in this clip.
`experiments/kalman_gridsearch.py` is therefore replay-only smoothing analysis.
It does not use `results/output.csv` as ground truth and does not report test
RMSE, position RMSE, or accuracy.

Default-config smoothing evidence:

| Metric | Value |
|---|---:|
| Frame-step std reduction | 59.9% |
| Second-difference std reduction | 84.4% |
| Waypoint-window raw radial std | 0.0877 m |
| Waypoint-window filtered radial std | 0.0869 m |
| Waypoint-window jitter reduction | 0.9% |

Replay grid search (`results/kalman_gridsearch_results.json`) reports 24
configs. The maximum smoothing config is `process_var=0.3`,
`measurement_var=0.1` with 74.1% frame-step reduction and 95.0% second-diff
reduction. The runtime keeps `3.0 / 0.01` because optimizing smoothing on one
clip without a validation set is single-scene fitting.

## Occlusion

The code path for detector dropouts is implemented:

1. BBox Kalman prediction keeps a track alive.
2. Lucas-Kanade optical flow propagates the bbox when detections disappear.
3. Stdout prints `OCCLUDED ... age=N` for predicted/held tracks.

Delivered-run evidence:

| Evidence | Value |
|---|---:|
| Real occluded rows in `results/output.csv` | 0 |
| Real LK-flow-assisted rows | 0 |
| Detector source | `blue_hsv_shape` for all 875 frames |

So the delivered output does not demonstrate a real person-occlusion event.
Continuity is only exercised by the forced detector-dropout stress suite in
`results/occlusion_stress_suite.json`.

This is a detector-dropout validation against the normal run, not hidden

## Jetson Orin NX Notes

The current submitted run is CPU/classical and uses no GPU. If replacing the
detector with a learned model on Jetson Orin NX:

| Topic | Deployment note |
|---|---|
| Quantization | Use TensorRT FP16 first; INT8 only with representative field calibration frames. |
| Toolchain | TensorRT is the NVIDIA path. RKNN is for Rockchip SoCs, not Orin NX. |
| Moving UAV | Replace fixed `R_cw,t_cw` with `T_world_body(t) @ T_body_cam` from the vehicle EKF. |
| IMU fusion | Fuse camera detections as delayed measurements with covariance, not as direct truth. |
| MAVLink | Send `LANDING_TARGET` for target-relative tracking or `VISION_POSITION_ESTIMATE` when publishing vehicle pose. |
| Rate | 10-20 Hz to the flight controller is realistic; publish covariance. |

Latency budget target on Orin NX:

| Stage | Budget |
|---|---:|
| Capture | 5-12 ms |
| Detect, TensorRT FP16 | 8-25 ms |
| Localize | <1 ms |
| Kalman update | <1 ms |
| MAVLink/UART transmit | 1-3 ms |
| End-to-end | 30-55 ms |

## Limitations

- The detector is scene-specific. It works on this clip; it is not a general
  outdoor trash-bin detector without additional data.
- No hidden bbox GT is available locally, so IoU > 0.6 cannot be claimed unless
  an external annotation file is supplied.
- No independent world-coordinate GT exists. Kalman analysis is smoothing and
  jitter analysis only.
- The delivered sequence has no real occlusion rows. Occlusion behavior is
  stress-tested with forced detector dropout only.
- `z_world` is imposed by the ground-plane model as `H/2 = 0.325 m`; sloped
  terrain or bottom-edge truncation will bias the estimate.
